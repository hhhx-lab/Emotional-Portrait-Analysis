# 情绪轮盘迭代版 SOP 摘要

本文件将《情绪轮盘画作解读工具-产品迭代文档.pdf》中与实现直接相关的要求整理为可执行规范。

## 核心工作流

1. 用户上传标准情绪轮盘作品照片。
2. Gemini 先做视觉观察，只输出结构化事实，不写心理结论。
3. Grok 再根据 Gemini 的观察结果撰写课程报告。
4. 服务端负责补齐确定性字段，例如标题、生成时间、昵称、统计数字和免责声明。

## 固定 8 分区

分区名称必须固定为：

- 愿望
- 温暖
- 希望
- 恐惧
- 杂乱
- 激动
- 期待
- 注视

所有模型输出、前端标签、示意图、统计逻辑都必须使用这一套名称，不允许回退到旧版的“未知 / 忽视”。

## Gemini 观察阶段

- 只写可见事实，不做心理推断。
- 输出顺序：整图判断 -> 质量判断 -> 8 分区证据 -> 关键元素 -> 不确定性 -> 补拍建议。
- 每个分区至少包含：`name + status + evidence`。
- `status` 只允许：
  - `painted`：有明确绘画或涂色痕迹
  - `blank`：明显留白
  - `unclear`：受画质、遮挡、角度等影响，无法稳定判断
- 不再设置独立中心区字段；中心附近内容只能融入整体观察或关键元素。

## Grok 报告阶段

- 只能消费 Gemini 结构化观察结果，不直接看图。
- 报告语气温暖、克制、非诊断。
- 禁止机械重复句式，例如“提示你”“提醒你”“反映出你”。
- 少用“营造出 / 传达出 / 强化了”这类艺术评论词，优先使用更自然的陪伴式表达。
- 必须承认识别边界，不得把不清晰区域写成确定事实。
- `recognition_note` 要短、透明、好理解。
- `action_suggestions` 是最高优先级，必须写成现实生活中的小行动，不能要求继续画、补色或补线。

## 最终报告结构

报告结构固定为：

1. `header`
2. `recognition_note`
3. `disclaimer`
4. `overall_impression`
5. `zone_insights`
6. `key_elements`
7. `comprehensive_insight`
8. `action_suggestions`
9. `closing`

## 文风要求

- 全文以自然段为主，不要像模板拼装。
- 长短句混合，避免千篇一律的相同开头。
- 允许心理教育式理解，但不能升级为诊断或治疗建议。
- 建议部分要写“接下来可以做什么”，而不是再次解释画面。
- UI 文案对用户不展示技术链路，不突出模型协作细节。
